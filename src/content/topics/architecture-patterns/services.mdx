Pick the wrong database and you can migrate off it in a quarter. Draw the wrong service boundaries and they follow you for years, because boundaries harden into team structures, API contracts, and data ownership that every future change must negotiate. How you split a system into services determines your delivery velocity more than any technology choice you will make — which is why the question deserves more rigor than "monoliths are legacy, microservices are modern."

## The spectrum

There are not two options but a spectrum, and the middle point is the most underrated.

A **monolith** is one deployable: one build, one release, one process (scaled horizontally as identical copies). Function calls between modules cost nanoseconds and cannot fail independently. One database means real ACID transactions across the whole domain — see ACID Transactions for the mechanics. Refactoring across module lines is an IDE operation, not a cross-team negotiation. The pain arrives with scale of *people*, not traffic: fifty engineers in one codebase contend for the release train, a memory leak anywhere restarts everything, and the hottest 5 percent of the code forces you to scale the other 95 percent with it.

A **modular monolith** keeps the single deployable but enforces internal boundaries: modules expose narrow interfaces, tooling fails the build when module A reaches into module B's internals, and each module keeps its tables private by convention. Shopify runs one of the largest Rails codebases in existence this way — millions of lines, over a thousand engineers, flash-sale traffic — as a "componentized" monolith rather than a service mesh.

**Microservices** make each boundary a process boundary: independent deploys, independent scaling, independent failure. The price is that every in-process call becomes a network hop (0.5–10 ms plus serialization, versus nanoseconds), every cross-service write becomes a distributed-consistency problem, every interface becomes a versioned contract you can never change unilaterally, and every service adds an on-call surface: discovery, tracing, dashboards, deploy pipeline.

| | Monolith | Modular monolith | Microservices |
|---|---|---|---|
| **Deploy** | one artifact, one pipeline | one artifact, one pipeline | pipeline per service, independent cadence |
| **Transactions** | real ACID across the domain | real ACID across the domain | distributed — sagas, eventual consistency |
| **Scaling** | whole app scales as one unit | whole app scales as one unit | each service sized to its own load |
| **Team autonomy** | shared codebase, merge contention | owned modules, shared release train | full — own repo, own deploys, own pace |
| **Ops cost** | one thing to run and watch | one thing to run and watch | discovery, tracing, contracts, on-call × N |

## You ship your org chart

Conway's law (1968): a system's structure mirrors the communication structure of the organization that builds it. It cuts both ways. Boundaries that cross team lines generate constant coordination overhead; boundaries that match team lines let each team move at its own speed. Amazon exploited this deliberately — two-pizza teams, each owning a service end to end ("you build it, you run it"), with the early-2000s API mandate forcing all inter-team access through service interfaces. The services worked because the teams were shaped first.

The practical heuristics for where to cut, in rough priority order:

1. **Bounded contexts** — DDD-lite: a boundary is good where the vocabulary changes. "Order" means different things to checkout and to fulfillment; that seam is real.
2. **Team ownership** — one service, one owning team. A service owned by three teams has three roadmaps and no owner.
3. **Rate of change** — code that changes together should live together. Splitting two modules that always change in lockstep converts every feature into a two-repo, two-deploy dance.
4. **Data ownership** — the decisive one, below.

## Data ownership is the real boundary

The rule: **each service owns its database, and no other service touches it**. Other services get the data through the owner's API or its published events (Sync vs Async covers those mechanics). The schema stays private, so the owner can refactor tables, add indexes, or shard (see Sharding and Partitioning) without asking anyone.

The corresponding anti-pattern is **integration through a shared database**: several "services" reading and writing the same tables. The schema becomes a frozen public contract — every migration needs sign-off from every reader, every hot query from one service degrades the others, and you have all the operational cost of separate deployables with none of the autonomy. Split processes that share a database are not services; they are a monolith with extra steps.

## The distributed monolith

The worst outcome on the spectrum is not the monolith — it is the **distributed monolith**: many deployables that must still be released together because their boundaries were drawn wrong. Symptoms: a feature routinely touches four repos; services share a database or fine-grained synchronous call chains; a deploy order document exists; one service's outage cascades through the fleet. You pay microservice costs — network latency on every call, contract versioning, an API Gateway and tracing stack, per-service on-call — while keeping monolith coupling. Every property in the comparison table lands in the worse column.

Segment lived this publicly. Their per-destination workers multiplied into 140-plus services and repos, shared libraries drifted out of sync across all of them, and autoscaling 140 load profiles consumed the team. In 2018 they collapsed the fleet back into a monolith and wrote up the retreat ("Goodbye Microservices"). The lesson is not "microservices bad" — it is that boundaries drawn along the wrong axis multiply, not divide, your operational load.

Other recurring pitfalls: the entity service ("UserService" that every request touches, a distributed single point of failure); nano-services so small that the network chatter dwarfs the work; and splitting before the domain is understood, which guarantees the bounded contexts are guesses.

## Deciding: start modular, extract on pain

Default to a modular monolith. It preserves the cheap options — real transactions, atomic refactors, one pipeline — while building the module seams that make later extraction a mechanical exercise instead of an archaeology project. Extract a module into a service only when you can name the specific pain, and the pain is one of these:

- **Independent scaling** — one module needs 10x the capacity or different hardware (a GPU inference path, a memory-hungry report generator) and scaling the whole artifact for it is wasteful.
- **Team contention** — teams block each other's releases; the deploy train is the bottleneck, not the code.
- **Fault isolation** — a defect in a peripheral module (PDF export) can take down a critical one (checkout), and process isolation is the fix.
- **Divergent constraints** — a different language or runtime is genuinely needed, or a compliance scope (payments, PII) is cheaper to audit in isolation.
- **The seam is already clean** — the module's data access is private and its interface is narrow. If it is not, fix the seam inside the monolith first; the network will not fix it for you.

If none of those apply, the honest answer to "should we do microservices?" is not yet — and the modular monolith means "not yet" never becomes "too late."
