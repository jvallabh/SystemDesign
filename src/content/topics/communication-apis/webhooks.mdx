Most integrations begin with the consumer asking the provider the same question on a loop: has anything changed since last time? Poll every 30 seconds and you burn about 2,880 requests per day per client — nearly all answered "no" — while a genuinely fresh event still waits up to 30 seconds to be noticed. Webhooks invert the relationship: the consumer registers an HTTPS URL, and the provider POSTs an event to it the moment something happens. Instead of you calling the API, the API calls you.

## Push, not poll

| | Polling | Webhooks |
|---|---|---|
| Notification latency | your poll interval (seconds to minutes) | typically under a few seconds |
| Cost at rest | constant requests even when nothing changes | zero traffic until an event occurs |
| Who owns delivery | consumer (retry your own polls) | provider (queue, retries, dead-letter) |
| Consumer infrastructure | a cron job or loop | a public, authenticated HTTPS endpoint |
| Failure mode | stale data, rate-limit pressure | missed or duplicated deliveries |

The trade is clear: webhooks buy low latency and low idle cost, and pay for it by turning the consumer into a server — one that must be reachable from the internet, verify who is calling, and survive duplicates. That last column is where most webhook integrations go wrong, so the rest of this page is about doing it right on both sides.

## The delivery lifecycle

A production webhook delivery is never "event happens, fire an HTTP request inline." The provider decouples the two with a queue, because the consumer's endpoint is slow, down, or deploying at any given moment, and a payment must not fail because a merchant's notification server did.

The steps, end to end: an event occurs and is written durably (often alongside the business transaction, via an outbox). A fan-out step finds every endpoint subscribed to that event type — one `order.paid` may become five deliveries. Each delivery is enqueued independently. A worker POSTs the event with an HMAC signature over the body, and waits a bounded time for a response — GitHub gives you 10 seconds before declaring the delivery dead. A 2xx marks it delivered; anything else, or a timeout, schedules a retry with exponential backoff. Stripe keeps retrying for up to three days, stretching intervals from minutes to hours; after the final failure the delivery is dead-lettered, the endpoint may be auto-disabled, and the consumer gets an email instead of a POST. (Queueing mechanics in general are the [Message Queues](/SystemDesign/topics/architecture-patterns/message-queues/) page's territory.)

## What the provider does and does not promise

Retries create the defining contract of webhooks: **at-least-once delivery**. If the consumer processed the event but the 2xx got lost in a timeout, the provider retries and the consumer sees the event twice. Exactly-once is not on offer — so every consumer must be idempotent (the [Idempotency](/SystemDesign/topics/distributed-systems/idempotency/) page covers the techniques; here, deduping by event id is usually enough).

Ordering is the second non-promise. Retries and parallel workers mean `invoice.paid` can arrive before the `invoice.created` it logically follows. Stripe says so explicitly. Consumers must key off the event's content and current state, never off arrival order.

Payload size is a design choice. **Fat payloads** embed the full object — convenient, but the data may be minutes stale by the time a third retry lands, and sensitive fields sit in the consumer's request logs. **Thin payloads** send only an event id and type; the consumer calls back to fetch the current object.

Fetch-on-receipt costs one extra API call per event, but it makes staleness impossible, keeps secrets behind normal API auth, and turns the webhook into what it really is: a hint that says "go look."

## The consumer's four rules

**Verify the signature.** A webhook endpoint is a public URL that mutates your system; without verification, anyone who guesses it can forge a `payment.succeeded` and ship themselves free goods. Providers sign each delivery with HMAC-SHA256 over the raw body using a shared secret — Stripe in the `Stripe-Signature` header, GitHub in `X-Hub-Signature-256`. Recompute the HMAC over the exact bytes received (before any JSON parsing, which can reorder or reformat) and compare with a constant-time function. Stripe's signature also covers a timestamp; reject anything older than about five minutes so a captured request cannot be replayed later. (GitHub's does not — its scheme relies on the secret alone.)

**Return 2xx fast.** The provider's timeout is your budget — respond in well under 10 seconds, ideally tens of milliseconds. Any real work done inline risks a timeout, which triggers a retry, which duplicates work you already did.

**Process asynchronously.** The handler should do exactly three things: verify, persist or enqueue, respond. Everything else happens in a worker after the ack. This is the [Sync vs Async](/SystemDesign/topics/architecture-patterns/sync-vs-async/) pattern applied at your front door.

**Dedupe by event id.** Store processed event ids (a unique-constraint insert is the simplest form) and skip repeats. This single table converts at-least-once delivery into effectively-once processing.

```text
POST /hooks/stripe            ← thin event evt_9f2, type payment.succeeded
  verify HMAC + timestamp     reject on mismatch (400)
  INSERT event id             duplicate? return 200, stop
  enqueue job                 respond 200 in ~20 ms
  ...worker fetches object, updates order, sends email
```

## Trade-offs and pitfalls

- **You inherit server problems.** Your endpoint must be up, fast, and reachable — a bad deploy now loses provider events, not just user requests. Dead-letter replay tooling on the provider side (Stripe's dashboard resend, GitHub's redelivery API) is your safety net; know where it is before you need it.
- **Webhooks are not a source of truth.** Deliveries can be dropped after retries are exhausted. Serious consumers pair webhooks with a periodic reconciliation poll — webhooks for latency, polling for completeness.
- **Silent failure is the default.** If your endpoint 200s while throwing away events, nobody notices for weeks. Monitor delivery age and event counts against expectations.
- **Local development friction.** The provider cannot reach `localhost`; you need a tunnel (Stripe's CLI forwards events for exactly this reason).
- **Skipped verification.** The most common real-world hole: teams ship the happy path and never check the signature. Treat an unverified webhook body as untrusted user input, because that is what it is.

## When to reach for it

Offer webhooks when you are the provider and consumers care about events measured in seconds, not minutes — payments, CI status, repository pushes. Consume them the same way, but budget for the full checklist: signature verification, fast ack, async processing, dedupe, and a reconciliation sweep. Stick with polling when events are rare and latency-tolerant (a nightly sync), when the consumer cannot expose a public endpoint, or when you need a guaranteed-complete listing rather than a best-effort stream. The strongest integrations use both: webhooks as the doorbell, the API as the source of truth.
