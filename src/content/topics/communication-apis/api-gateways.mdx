Split a backend into twenty services and you inherit a question the monolith never asked: what do clients actually connect to? Exposing every service publicly means twenty places to terminate TLS, validate tokens, enforce quotas, and log requests — and clients that must know your internal topology, so every service split breaks someone. An API gateway answers with one front door: a single edge layer that every external request passes through before anything internal sees it.

## What the front door does

A gateway is a reverse proxy that has been taught about APIs. On every request it can perform some mix of:

- **Routing** — map `/orders/*` to the orders service and `/users/*` to the users service, by path, host, or header.
- **Authentication and authorization** — verify the JWT or API key once at the edge, reject bad credentials before they consume backend capacity, and pass a trusted identity header downstream so services never re-parse tokens (token mechanics live in the [JWTs](/SystemDesign/topics/communication-apis/jwts/) topic).
- **Rate limiting and quotas** — per API key or per user, enforced before the request touches a database (algorithms are the [Rate Limiting](/SystemDesign/topics/scalability/rate-limiting/) topic's job).
- **TLS termination** — certificates live in one place; internal traffic can run plaintext or mesh-managed mTLS.
- **Request/response transformation** — rewrite paths, inject headers, strip internal fields from responses, version-shim old clients.
- **Protocol translation** — accept HTTP/JSON from browsers, speak gRPC to internal services; or bridge [WebSocket](/SystemDesign/topics/communication-apis/long-polling-vs-websockets/) connections to backend pub/sub.
- **Response aggregation** — fan out one client call to several services and merge the results.
- **Observability** — one choke point that logs, meters, and traces 100% of external traffic with consistent request IDs.

Kong (built on nginx), Envoy-based gateways such as Emissary and Gloo, AWS API Gateway as a managed option, and hand-rolled nginx configs all occupy this slot.

## Why it exists: cross-cutting concerns, implemented once

Every capability above is a *cross-cutting concern*: every service needs it, and none of it is any service's business logic. Without a gateway, each of twenty teams implements auth, quotas, and TLS twenty times, in several languages, drifting apart in subtle ways — and patching a TLS library vulnerability or changing the auth scheme means twenty coordinated deploys. With a gateway, it is one implementation, one config, one deploy. Security teams especially like this: the attack surface has one owner.

The second payoff is a **stable public surface over shifting internals**. Clients see `/api/v1/orders`; behind the gateway you can split the orders service in two, migrate it from REST to gRPC, or move it to another cluster, and no client notices. The gateway is the seam that lets internals move.

## Gateway vs reverse proxy vs load balancer vs service mesh

These four get conflated because they all sit between callers and servers. They differ in direction and job:

| Layer | Traffic direction | Core job | Distinct because |
| --- | --- | --- | --- |
| Reverse proxy | north-south | forward requests to backends | generic building block; a gateway is a reverse proxy plus API-aware policy (keys, quotas, transformation) |
| Load balancer | north-south | spread load across replicas of *one* service | picks *which copy*; a gateway picks *which service* |
| API gateway | north-south | one policy-enforcing front door across *many* services | API-level semantics: routes, auth, per-consumer limits |
| Service mesh | east-west | secure and observe service-to-service calls via sidecars | manages traffic *between* services, not from the outside world |

In practice they compose: an L4 load balancer spreads traffic across gateway replicas, the gateway routes to a service, and an internal balancer or mesh sidecar picks the replica. Proxy fundamentals belong to the [Proxy vs Reverse Proxy](/SystemDesign/topics/scalability/proxy-vs-reverse-proxy/) topic, and balancing algorithms to [Load Balancing](/SystemDesign/topics/scalability/load-balancing/).

## Backends for frontends

One general-purpose gateway serves every client equally — which often means equally badly. A mobile app on a flaky cellular link wants one round trip and a 5 KB payload; the web app wants rich nested data; partners want a frozen, versioned contract. The **BFF pattern** (backend for frontend) gives each client class its own small gateway, owned by the team that builds that client. The mobile BFF calls users, orders, and pricing internally, then returns exactly the compact shape the mobile screen needs — aggregation moves server-side, where inter-service hops cost a millisecond instead of a hundred. Netflix's edge API layer evolved this way for the same reason: hundreds of device types, no single response shape that fit them all.

Keep BFFs thin: they shape responses for one client, they do not own business rules. Two BFFs re-implementing checkout logic is the microservice version of copy-paste.

## Pitfalls

**Business logic creep.** The gateway's plugin system makes it temptingly easy to add "just one check" — and a year later, order validation lives in Lua snippets inside gateway config, untested and owned by nobody. Rule of thumb: the gateway decides *whether and where* a request goes; services decide *what happens*. If a change needs domain knowledge, it belongs in a service.

**Single point of failure, plus a hop.** Every request now pays an extra network hop — typically 0.5–2 ms in-region, rarely the problem — but when the gateway is down, *everything* is down. Treat it as tier-zero infrastructure: stateless, replicated behind a dumb L4 balancer, config rollouts canaried like any deploy. A bad gateway config push is a full outage, not a partial one.

**The gateway team as bottleneck.** If every new route or header tweak needs a ticket to a central platform team, the gateway becomes an organizational choke point that slows every product team. The fix is self-service: per-team declarative route config (Kubernetes's Gateway API model, or per-service config files merged at deploy) with the central team owning the platform, not each route.

## When to reach for it

A monolith with one client needs nginx, not a gateway product. Reach for a real gateway when you have several services behind one public API, external consumers needing keys and quotas, or a protocol boundary (public REST, internal gRPC). Add BFFs when distinct client classes genuinely want different response shapes — not before, because each BFF is another deployable to own. And do not confuse the axes: a gateway will not fix east-west observability, and a service mesh will not give you a public front door. North-south goes through the gateway; east-west goes through the mesh.
