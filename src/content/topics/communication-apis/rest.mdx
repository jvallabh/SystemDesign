Every network API has to answer the same questions: how do clients name things, how do they change them, and how can caches, proxies, and load balancers help without understanding your application? REST is not a protocol or a spec — it is Roy Fielding's 2000 answer to those questions, a set of constraints that, when followed, let an API inherit the entire machinery of the web for free.

## Constraints, and what each one buys

Fielding derived REST by adding constraints one at a time and noting what property each purchases:

- **Client-server** — the UI and the data storage evolve independently. Your iOS app and your database ship on different schedules; the interface between them is the contract.
- **Stateless** — every request carries everything needed to process it. The server keeps no conversation memory between requests. This is the constraint that pays the biggest dividends (more below).
- **Cacheable** — responses declare whether they may be reused. A `GET` with `Cache-Control: max-age=300` can be answered by a browser cache, a corporate proxy, or a [CDN](/SystemDesign/topics/caching/cdn/) edge without your server ever seeing it.
- **Uniform interface** — the same small verb set applies to every resource. An intermediary that has never heard of your app still knows `GET /orders/42` is a safe read it may cache and retry.
- **Layered system** — a client cannot tell whether it is talking to the origin or to a proxy three hops away. That opacity is precisely what lets you insert a CDN or an [API gateway](/SystemDesign/topics/communication-apis/api-gateways/) later without touching a single client.

A sixth constraint, **code-on-demand** (servers shipping executable code to clients), is explicitly optional and rarely relevant to API design.

## Resources and representations

REST models the world as **resources** — nouns with stable URLs — and exchanges **representations** of them, usually JSON today. The resource `/users/91` is an abstract thing; the JSON document you receive is one representation of it, negotiated via `Accept` headers. Design starts by finding the nouns:

```
GET    /orders              list orders
POST   /orders              create an order
GET    /orders/42           read one
PUT    /orders/42           replace it
PATCH  /orders/42           update part of it
DELETE /orders/42           remove it
GET    /orders/42/items     line items of order 42
```

Actions that resist noun-ification usually hide a resource. "Cancel an order" is not `POST /orders/42/cancel`; it is `PATCH /orders/42` setting `status: "cancelled"` — or, when the action has its own lifecycle, a new resource: `POST /orders/42/cancellations` creates a cancellation you can later inspect. Stripe models refunds this way: a refund is a first-class object with an ID, not a verb fired at a charge.

## Verb semantics: safety and idempotency

The verbs carry machine-readable promises. **Safe** means no state change — intermediaries may prefetch and cache freely. **Idempotent** means repeating the request yields the same end state — clients and proxies may retry on timeout without fear of double effects.

| Method | Safe | Idempotent | Typical use |
|---|---|---|---|
| GET | yes | yes | read a resource |
| HEAD | yes | yes | headers only (existence, size) |
| PUT | no | yes | full replace at a known URL |
| DELETE | no | yes | remove; repeat is a no-op |
| PATCH | no | no by default | partial update |
| POST | no | no | create, or anything non-idempotent |

These promises are load-bearing. A [load balancer](/SystemDesign/topics/scalability/load-balancing/) can transparently retry an idempotent `PUT` against another replica after a timeout; retrying a `POST` might charge a card twice, which is why payment APIs bolt on [idempotency keys](/SystemDesign/topics/distributed-systems/idempotency/) to recover the property POST lacks.

## Statelessness is the scaling constraint

Because no request depends on server-side conversation state, any replica can serve any request. Add a tenth server and the load balancer can use it immediately — no session affinity, no sticky routing, no state to migrate when a server dies. Client identity travels in the request itself (a bearer token, covered in depth under [JWTs](/SystemDesign/topics/communication-apis/jwts/)), and durable state lives in the database. The broader trade-off has its own page, [Stateful vs Stateless](/SystemDesign/topics/scalability/stateful-vs-stateless/); here it is enough to see that statelessness is what makes "just add more servers" true.

## Status codes are part of the contract

The response code is the machine-readable verdict; the body is commentary. The families matter more than memorizing all 60-odd codes: 2xx success (`201 Created` with a `Location` header, `204 No Content` for deletes), 3xx redirection (`304 Not Modified` powers conditional caching), 4xx client fault (`400` malformed, `401` unauthenticated, `403` unauthorized, `404` absent, `409` conflict, `429` rate-limited), 5xx server fault. Retry logic, monitoring dashboards, and CDN behavior all key off these numbers — a `503` gets retried, a `404` gets cached-as-missing, a `400` gets neither.

## HATEOAS and the maturity ladder

Fielding's full vision — hypermedia as the engine of application state — has responses embed links to legal next actions, so clients navigate the API the way a person navigates a website, discovering URLs rather than hardcoding them. Almost nobody implements it fully: generated client SDKs and OpenAPI docs already solve discoverability, and few teams will spend response bytes and client complexity on link-following logic. The Richardson maturity model names the rungs, and level 2 is where nearly every production API — GitHub, Stripe, Twilio — deliberately stops (GitHub includes hypermedia links, but its clients still hardcode URLs).

## Pragmatic conventions

The industry converged on a dialect that the original dissertation never mentions:

- **Plural nouns**: `/orders`, not `/order` — one grammar for collections and members.
- **Shallow nesting**: `/orders/42/items` is fine; three levels deep is a smell. If an item has a global ID, expose `/items/7` directly.
- **Filtering and sorting via query string**: `GET /orders?status=shipped&sort=-created_at`.
- **Pagination always**, from day one: cursor-based (`?cursor=abc&limit=50`) survives concurrent inserts; offset-based breaks under them. Stripe's `starting_after` cursors are the reference design.
- **Versioning**: a URL prefix (`/v2/orders`) is crude but visible and cacheable; header-based versioning is purer and nearly invisible in logs. Pick one; general contract-evolution strategy belongs to the [APIs](/SystemDesign/topics/communication-apis/apis/) page.

## Common mistakes

- **Verbs in URLs.** `POST /getUserData` throws away the uniform interface — no cache, proxy, or engineer can infer anything from it. That is RPC wearing REST's clothes.
- **200 with an error body.** Returning `200 OK` wrapping `"success": false` blinds every layer that speaks HTTP: caches store the failure, monitors report green, retry logic never fires. The status code is the contract.
- **Chatty resource design.** Rendering one screen should not take 15 sequential GETs; at 50ms per round trip that is 750ms of pure waterfall. Design coarse resources around read patterns, or offer expansion (`?expand=customer`) — this under-fetching problem is the itch GraphQL was built to scratch, compared properly on the [API-styles page](/SystemDesign/topics/communication-apis/rest-grpc-graphql-trpc/).
- **Non-idempotent retries.** If a `POST` can be retried by a client on timeout, accept an idempotency key or you will eventually double-charge someone.

## When to reach for it

REST won because it rides infrastructure that already exists: every firewall passes it, every CDN caches it, every language ships an HTTP client, every engineer already speaks it, and `curl` is your debugger. Default to REST for public APIs, CRUD-shaped domains, and anything third parties integrate with. Look elsewhere when you need streaming or the lowest possible latency between internal services, or when clients need to shape their own responses — the trade-offs live in REST vs gRPC vs GraphQL vs tRPC.
