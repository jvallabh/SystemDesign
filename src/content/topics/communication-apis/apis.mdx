Two pieces of software written by different people, in different languages, deployed on different continents, still need to cooperate. The only way that works is a contract: a fixed, documented surface that one side promises to serve and the other side codes against. An API is that contract — and almost everything that goes wrong with APIs comes from treating it as an implementation detail instead of a promise.

## Interface vs implementation

The idea predates networks. A function signature is an API: callers depend on the name, parameters, and return type, and the body can be rewritten freely as long as those hold. A network API is the same separation with a wire in the middle — the contract is now URLs, methods, schemas, and status codes, and the implementation behind it can move from a Python monolith to a fleet of Go services without any client noticing. That decoupling is the entire value: it lets teams, companies, and release schedules vary independently on each side of the boundary.

## Anatomy of a good contract

A usable contract specifies far more than endpoints. Each piece removes a class of guesswork:

- **Operations** — what you can do, and which calls are safe to retry. Read operations should be side-effect free; retryable writes need idempotency keys (the [Idempotency](/SystemDesign/topics/distributed-systems/idempotency/) page covers the mechanics).
- **Schemas** — exact field names, types, nullability, and units. "amount" is ambiguous; "amount in minor currency units, integer" is a contract.
- **Errors** — every failure mode a client must handle, with machine-readable codes.
- **Pagination** — how large collections are traversed, guaranteed on every list endpoint.
- **[Rate limits](/SystemDesign/topics/scalability/rate-limiting/)** — the request budget, plus how the API says "slow down" (429 with a `Retry-After` header) so clients can back off instead of hammering.
- **Compatibility rules** — which changes the provider may make without notice, and which require a new version.

The transport style — REST conventions, gRPC, GraphQL — is a separate decision covered by the [REST](/SystemDesign/topics/communication-apis/rest/) page and the [API-styles comparison](/SystemDesign/topics/communication-apis/rest-grpc-graphql-trpc/); every one of them still needs all six items above.

## Error design

A status code alone is not an error contract. `400 Bad Request` tells a client something failed; it does not say which field, why, or whether retrying helps. Good APIs pair the status code with a machine-readable body:

```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "amount_too_small",
    "param": "amount",
    "message": "Amount must be at least 50 cents.",
    "doc_url": "https://docs.example.com/errors#amount_too_small"
  }
}
```

The status code routes coarse behavior (4xx: fix the request, except 429, which means back off; 5xx: retry with backoff), the `code` field drives programmatic handling, and the message is for humans and logs — never for `if` statements, because you will want to reword it later. RFC 9457 ("problem details") standardizes this shape if you would rather not invent one. The most common failures are returning 200 with an error buried in the body — which defeats every retry proxy and monitoring rule that keys on status — and stuffing prose into a single string that clients end up regex-matching.

## Pagination

Any endpoint that lists things needs pagination from day one, because retrofitting it is a breaking change. Two patterns dominate:

| | Offset (`?offset=200&limit=100`) | Cursor (`?starting_after=obj_123`) |
|---|---|---|
| Mechanics | Skip N rows, return the next page | Return rows after an opaque position token |
| Jump to page 47 | Yes | No — forward/backward only |
| Stability under writes | Rows shift: items skipped or repeated | Stable — the cursor pins a position |
| Cost on large tables | Degrades: the database still scans the skipped rows | Constant with an index on the sort key |
| Typical use | Admin UIs, small static datasets | Feeds, sync jobs, anything large or live |

Offset also rots at scale: `OFFSET 100000` forces the database to read and discard 100,000 rows to serve one page. Cursors dodge both problems, which is why Stripe, Slack, and Twitter's APIs are cursor-based. Keep the cursor opaque — a base64 token, not a visible ID — so you can change its encoding later.

## Versioning and why breaking changes cost so much

Hyrum's law: with enough users, every observable behavior of your API will be depended on by somebody — the field order, the exact error message, the undocumented timeout. So a "breaking change" is anything a client can observe, not just what you documented, and shipping one means coordinating a migration across code you do not control and cannot even see. For a public API that can take years: Stripe has kept decade-old pinned API versions working for clients that never upgraded.

| Strategy | How | Trade-off |
|---|---|---|
| URL versioning | `/v2/orders` | Visible and cacheable, but v2 is a fork — every change now lands twice |
| Header versioning | `Stripe-Version: 2024-06-20` per account | Clean URLs, per-client pinning; invisible in logs and curl unless you look |
| Additive-only evolution | Never remove or change fields; only add optional ones | No version machinery at all, but the schema accretes forever and mistakes are permanent |

Most durable APIs combine the last two: evolve additively by default, and reserve version bumps — pinned per client, defaulted for new signups — for the rare change that cannot be additive. Whatever you choose, publish the rules: clients must know that new optional fields may appear at any time, so their parsers must ignore unknown keys.

## What separates great APIs from mediocre ones

Stripe and Twilio are the usual quality bar, and the reason is boring discipline, not cleverness. Every Stripe list endpoint paginates identically; every object carries `id`, `object`, and `created`; every error follows one shape. That consistency means learning one endpoint teaches you all of them — the principle of least surprise applied ruthlessly. The second half is documentation as a first-class deliverable: an OpenAPI specification that is generated from (or validated against) the real implementation, runnable examples in several languages, and a changelog that treats every behavioral change as news. Stripe's published spec doubles as the source for its docs and client SDKs, so none of them can drift from reality.

## Pitfalls

- **Leaking internals.** Exposing database column names or internal enums welds your schema to your storage; renaming a column becomes a breaking API change.
- **Errors as prose.** If the only signal is a message string, clients will parse it, and rewording it becomes a breaking change (Hyrum again).
- **Offset pagination by default.** Fine at 1,000 rows, a duplicate-riddled table scan at 10 million.
- **Booleans that should be enums.** `status: "pending"` can grow new states additively; `is_complete: false` cannot.
- **Versioning as an afterthought.** If v1 shipped with no compatibility rules, every client assumption is now load-bearing.
- **Undocumented limits.** Clients that never see a rate-limit contract discover it as a production outage.

## When to invest

Match the contract's rigor to the cost of changing it. A public API is effectively permanent — spec it first (OpenAPI), review the spec like code, and get pagination, error shapes, and compatibility rules right before the first external caller, because after that every fix is a migration. Internal service-to-service APIs deserve the same shapes but can version faster, since you can find and update every caller. The one habit that pays everywhere: design the contract before the implementation, and treat any observable change as a change to the contract — because to your clients, it is.
