A busy database runs your code in thousands of overlapping interleavings, any of which can be cut short by a crash. Reasoning about every interleaving and partial failure by hand is intractable — so the database sells you a contract instead. That contract is ACID: wrap statements in a transaction and, at its strongest settings, they execute as if alone and as if instantaneous, whatever else is running or failing.

## Atomicity: all or nothing

A transfer of $100 between accounts is two writes: debit A, credit B. If the server dies between them, $100 has vanished. Atomicity guarantees that either both writes survive or neither does — there is no observable in-between, even across a crash.

The mechanism is logging. Before touching any data page, the engine appends the change to a **write-ahead log (WAL)** — a sequential file that is cheap to fsync. Commit means "the log record is durably on disk"; the actual data pages are updated lazily, often seconds later. After a crash, recovery replays the log: committed transactions are redone, uncommitted ones rolled back via undo records. Atomicity is simply the ability to reconstruct a clean state from the log.

## Consistency: the odd one out

Consistency here means "application-defined invariants hold": balances never go negative, every order row has a customer. Unlike the other three letters, the database cannot supply this on its own — it knows nothing of your invariants beyond declared constraints (foreign keys, CHECK, UNIQUE). Consistency is a contract: if each transaction individually preserves the invariants, atomicity, isolation, and durability guarantee the database as a whole stays valid. Joe Hellerstein has quipped that the C was tossed in mainly to make the acronym work. (It is unrelated to [CAP](/SystemDesign/topics/distributed-systems/cap-theorem/)'s consistency, which is about replica agreement.)

## Isolation: the deep one

Isolation is where the real engineering lives. Perfect isolation — **serializability** — means the outcome equals some one-at-a-time execution of the transactions. But enforcing it costs throughput, so every database offers weaker levels that permit specific, named anomalies:

- **Dirty read** — reading another transaction's uncommitted (possibly rolled-back) write.
- **Non-repeatable read** — reading the same row twice inside one transaction and getting different values.
- **Phantom** — re-running a query and seeing rows appear or vanish because another transaction inserted or deleted matches.
- **Write skew** — two transactions each read data, make a decision based on it, and write to *different* rows; each write is fine alone, but together they break an invariant.

| Isolation level | Dirty read | Non-repeatable read | Phantom | Write skew |
| --- | --- | --- | --- | --- |
| Read uncommitted | possible | possible | possible | possible |
| Read committed | blocked | possible | possible | possible |
| Repeatable read / snapshot | blocked | blocked | mostly blocked | possible |
| Serializable | blocked | blocked | blocked | blocked |

Write skew is the one that surprises people, because snapshot isolation — which feels airtight — permits it. Classic example: a hospital requires at least one doctor on call, and Alice and Bob both are. Each opens a transaction, reads "2 doctors on call", concludes it is safe to leave, and sets their *own* row to off-call. Neither touched the row the other wrote, so there is no write conflict — both commit, and zero doctors are on call. Only serializable isolation (or an explicit `SELECT ... FOR UPDATE` on the rows read) catches it.

### MVCC: how readers avoid blocking writers

Modern engines (Postgres, MySQL InnoDB, Oracle) implement isolation with **multi-version concurrency control** rather than read locks. Every write creates a new version of the row tagged with the writing transaction's ID; old versions stick around. A reader is handed a snapshot — "versions committed before I started" — untouched by concurrent writes. Readers never block writers and writers never block readers, which is why a long analytics query does not freeze OLTP traffic. The price: dead versions must be garbage-collected (Postgres VACUUM), and long transactions pin old versions, bloating tables.

### What your database actually gives you

Mainstream relational databases do not default to serializable because it costs real throughput: either pessimistic locking (readers and writers block each other) or Postgres-style serializable snapshot isolation, which tracks read/write dependencies and aborts transactions forming a dangerous cycle — forcing you to build retry loops. So defaults are weaker: **Postgres, Oracle, and SQL Server default to read committed; MySQL InnoDB defaults to repeatable read.** Postgres's "repeatable read" is actually snapshot isolation, which also prevents most phantoms — stronger than the SQL standard requires.

## Durability: surviving the crash

Durability means a committed transaction survives power loss. Mechanically it is the same WAL: commit does not return until the log record is fsynced — roughly 1 ms on an SSD, amortized by batching concurrent commits into one fsync (group commit). But a single disk can die outright, so serious deployments extend durability with replication: Postgres synchronous replication withholds the commit ack until a standby also has the WAL record. Some shops deliberately relax this (`synchronous_commit = off`), accepting the loss of the last few hundred milliseconds of commits for lower latency.

## Across machines: 2PC and sagas

Inside one node, ACID is a solved problem. Across services with separate databases it gets ugly. The classic answer, **two-phase commit**, has a coordinator ask every participant to *prepare* (make the transaction durable but undecided), then tells everyone to commit. The flaw is blocking: a prepared participant must hold its locks until the coordinator delivers a verdict. If the coordinator crashes after phase one, participants sit in doubt — rows locked, potentially for minutes. That is why 2PC rarely crosses service boundaries in modern architectures.

The microservices alternative is the **saga**: a chain of ordinary local transactions, each with a compensating action (refund the charge, restock the item) that runs if a later step fails. You trade isolation — intermediate states are visible — for availability, and every step must be safely retryable (the [Idempotency](/SystemDesign/topics/distributed-systems/idempotency/) topic covers that).

## Contrast: BASE

The opposite corner of the design space is BASE — Basically Available, Soft state, Eventual consistency — the philosophy behind Dynamo-style stores like Cassandra. BASE systems accept writes optimistically and converge replicas later, pushing conflict handling onto the application. It is not "ACID but worse"; it is a different bet — availability and write throughput over invariant enforcement — fitting shopping carts better than ledgers.

## Pitfalls

- **Assuming the default isolation level is serializable.** It never is. Check-then-write invariants (unique usernames, balance floors, on-call minimums) need serializable, an explicit lock, or a declared constraint.
- **Transactions held open across network calls.** An HTTP call inside a transaction pins locks and MVCC versions for the full round trip; do slow work outside.
- **No retry loop under serializable.** Serialization failures are normal operation, not errors — code that does not retry fails randomly under load.
- **Trusting ORM transaction boundaries.** Many frameworks auto-commit per statement unless told otherwise, quietly discarding atomicity.

## When to reach for it

If your invariants live inside one database, use transactions liberally at the default level — they are cheap — and escalate to `SELECT ... FOR UPDATE` or per-transaction serializable only for check-then-write operations. If an invariant spans services, do not reach for 2PC; restructure it as a saga, or redesign so the invariant lives in one place. And when someone proposes relaxing isolation or durability for speed, make them name the anomaly they are accepting — the table above is the checklist.
