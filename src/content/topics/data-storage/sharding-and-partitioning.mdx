A single database node has two hard ceilings: how much data fits on its disks, and how many writes per second its one primary can absorb. Read replicas and caching relieve read pressure, but neither adds a byte of write capacity or storage. When a table heads past a few terabytes, or sustained writes climb into the tens of thousands per second, the only move left is to split the data itself across machines. That split is partitioning, and every hard problem in it flows from one decision: *which rows live where*.

## Two axes of splitting

**Vertical partitioning** splits by table or column: move `orders` and `analytics_events` into separate databases, or peel large blob columns away from the hot row data they slow down. It's cheap and often the right first move, but it has a built-in ceiling — your biggest table still lives, whole, on one node.

**Horizontal partitioning — sharding** — splits the *rows* of one table across nodes, using a **shard key** derived from each row. There is no ceiling: more data or more writes means more shards. Everything below is about horizontal sharding, because that's where the sharp edges are.

## Range, hash, and directory

Every sharding scheme answers one question: given a key, which shard? The two core answers fail in exactly opposite ways.

| | Range | Hash | Directory |
|---|---|---|---|
| **Placement** | Contiguous key spans: shard 1 gets `a–h`, shard 2 gets `i–q`… | `hash(key)` decides; adjacent keys scatter | Explicit lookup table maps each key (or bucket) to a shard |
| **Range scans** | Fast — one shard, sequential read | Impossible without querying every shard | Depends on how buckets are assigned |
| **Load spread** | Only as good as your key distribution | Near-uniform by construction | Whatever you make it |
| **Failure mode** | **Hot ranges**: sequential keys (timestamps, auto-increment IDs) funnel every write to the newest shard | **No locality**: "last 24 hours of events" becomes a scatter-gather across all shards | The directory is an extra hop and a single point you must scale and keep available |
| **Used by** | HBase, Bigtable, CockroachDB | Cassandra, DynamoDB | Vitess topology, MongoDB config servers |

The range failure mode is the one people walk into: timestamp keys written a millisecond apart are neighbors, so every write happening right now lands on the same shard. You bought ten machines and one does all the work.

## The shard key: the decision you can't take back

The shard key gets baked into every query, every index, and the physical location of every row. Changing it later means rewriting the entire dataset while serving traffic — teams routinely spend quarters on it. Judge a candidate key on three criteria:

1. **Cardinality.** Enough distinct values to spread across far more shards than you'll ever have. `country` (about 200 values, wildly skewed) fails; `user_id` (millions) passes.
2. **Write distribution.** Load should spread across key values *at any instant*, not just in aggregate.
3. **Query alignment.** Your dominant queries should include the key, so they route to one shard.

Worked example — a `posts` table for a social app. `created_at` looks tempting but fails criterion 2 catastrophically: at any moment, all inserts carry roughly *now* as their key, so under range sharding one shard takes 100% of writes; hashing the timestamp fixes writes but then "this user's recent posts" hits every shard. `user_id` wins on all three: millions of values, writes spread across whoever's posting, and the dominant query — fetch one user's posts — lands on one shard, where a local index on `(user_id, created_at)` makes the recency scan cheap. This is why Instagram sharded Postgres by user, and why DynamoDB pushes you to query through its partition key.

## Hot keys

A good key distributes *keys* evenly; it can't make the traffic behind each key even. One celebrity account with 100M followers generates orders of magnitude more reads and fan-out writes than a median user — all landing on that key's shard. DynamoDB makes the ceiling concrete: each physical partition serves at most 3,000 read units and 1,000 write units per second regardless of the table's total throughput. A viral key throttles at that cap while the rest of the fleet idles.

Mitigations, in escalating order of effort:

- **Cache in front.** A hot key is by definition cacheable; a single Redis node absorbing reads for the top 0.1% of keys often ends the problem.
- **Salting / key splitting.** Rewrite the hot key `celeb42` as `celeb42#0` through `celeb42#15`, spreading its rows over 16 shards. Writes pick a random suffix; reads fan out to all 16 and merge. That trades one hot shard for 16x read amplification — apply it only to keys measured hot.
- **Isolate.** Move the key to dedicated hardware via a directory entry. Crude, effective, operationally annoying.

## Cross-shard pain

Everything a single-node database does across rows gets expensive when the rows live on different machines.

**Scatter-gather queries.** Any query that omits the shard key must be sent to all N shards and merged. Latency becomes the *slowest* shard's response — with 100 shards, the chance every shard beats its own p99 is 0.99^100, about 37%, so nearly two-thirds of scatter queries eat at least one shard's worst-1% latency. Throughput suffers too: one logical query consumes N shards' worth of work.

**Distributed transactions.** A transaction spanning shards needs two-phase commit — extra round trips and locks held across them (the ACID page covers 2PC and why it blocks). Most sharded systems forbid cross-shard transactions or make them slow enough that schemas contort to keep each transaction on one shard — e.g., colocating a user's orders with the user.

**Secondary indexes.** A *local* index lives on each shard and only covers that shard's rows, so querying by a non-key column is a scatter-gather. A *global* index is itself sharded by the indexed column — reads become targeted, but now every write updates two shards, which is why DynamoDB's global secondary indexes are updated asynchronously and read slightly stale.

## Resharding

Load grows; shard counts must change. Two strategies keep that survivable:

**Pre-split into many logical shards.** Create far more logical shards than physical machines on day one — thousands, not tens — and map many-to-one onto hardware. Growing capacity then means *moving whole logical shards*, never re-partitioning data. Instagram ran thousands of logical shards as Postgres schemas across a few dozen machines, and embedded a 13-bit shard ID inside every 64-bit post ID so any ID is self-routing forever. Vitess reshards by splitting a shard's keyspace range in two, copying rows and tailing the replication log until the new shards catch up, then cutting over in seconds. DynamoDB does the same invisibly, splitting any partition that passes roughly 10 GB or its throughput cap.

**Consistent hashing** attacks the other half of the problem — making `hash(key) mod N` stop reshuffling nearly every key when N changes — and has its own page; the short version is that adding one node should move about 1/N of the keys, not all of them.

## The routing tier

Something must translate `user_id 8412` into "shard 7, currently on host db-23." Three common homes for that logic:

- **Smart client** — the shard map ships inside the application (or is derivable, as with Instagram's embedded shard IDs). Zero extra hops, but every client must learn topology changes.
- **Proxy tier** — a stateless router like Vitess's `vtgate` or MongoDB's `mongos` speaks the database protocol, consults the shard map, and forwards. Clients stay dumb; you add one in-datacenter hop, well under a millisecond.
- **Coordinator nodes** — every database node can route, forwarding queries it doesn't own, as Cassandra does.

In all three, the map lives in a small strongly-consistent store — etcd, ZooKeeper, or a config database — and routers cache it. During a shard move the map marks the range as migrating so routers can double-write or redirect.

## Trade-offs and pitfalls

- **Sharding too early.** You permanently give up cheap joins, transactions, and unique constraints across the dataset, and take on a routing tier and per-shard migrations. A well-tuned single primary with replicas handles more than most teams expect.
- **Sharding by time.** The most common self-inflicted hot shard. If you must keep time-locality, compound the key — hash prefix first, time second — so writes spread but per-entity scans stay ordered.
- **Too few shards.** Doubling 4 physical shards means splitting live data. Pre-split logically; moving a whole logical shard is a file copy, splitting one is a migration project.
- **Unmonitored key skew.** Uniform hashing does not mean uniform load. Track per-shard write throughput and per-key heat from day one; celebrities appear without warning.
- **ORMs that silently scatter.** A query missing the shard key doesn't fail — it fans out to every shard and gets slow at the tail. Make scatter-gather loud: log it, alert on it, or require an explicit opt-in flag.

## When to reach for it

Exhaust the cheaper levers first: vertical scaling, read replicas, caching, vertical partitioning, archiving cold data. Reach for sharding when the *write path or raw size* — not reads — exceeds one machine: sustained writes beyond what a single primary commits comfortably, or a dataset one node can't store, back up, or restore in acceptable time (a multi-terabyte table whose restore takes a day is an availability problem). When you do: derive the shard key from your dominant access pattern, pre-split into many logical shards, and put the routing map in one authoritative place. The scheme you pick is the easy part; the key you pick is forever.
