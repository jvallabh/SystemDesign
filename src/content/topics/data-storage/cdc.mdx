Every serious system grows read-side copies of its database: a Redis cache, an Elasticsearch index, a warehouse table, another team's local view. The database is the source of truth, but four other systems need to reflect it — and the obvious way to keep them in sync, having the application write to all of them, fails in ways that are quiet, rare, and permanent. Change Data Capture flips the approach: instead of trusting the application to announce its writes, you tail the database's own record of what changed.

## The dual-write problem

Suppose an order service updates Postgres and then updates the search index:

```text
1. UPDATE orders SET status = 'shipped' WHERE id = 42;   -- succeeds
2. POST /orders/_update/42  { "status": "shipped" }       -- times out
```

Now the database says shipped and search says pending, and nothing will ever reconcile them. Swap the order and you get the mirror-image bug: search updated, then the database transaction rolls back, and search advertises a state that never existed. Wrap both in a try/catch and you still lose — the process can crash between the two calls. Two independent systems cannot be updated atomically without distributed transactions, which neither Redis nor Elasticsearch speaks.

There is a second, subtler failure: ordering. Two app instances update the same row concurrently; the database serializes them one way, but their follow-up cache writes race and land the other way. The cache now holds the older value forever.

## Three ways to capture changes

| | Query polling | Triggers | Log-based |
|---|---|---|---|
| Mechanism | Re-query rows where `updated_at` is newer than the last scan | Trigger writes each change to an audit table | Tail the WAL / binlog the database already writes |
| Deletes | Missed — the row is gone | Captured | Captured |
| Intermediate states | Missed if a row changes twice between polls | Captured | Captured |
| Latency | Poll interval (seconds to minutes) | Near-real-time | Near-real-time (typically well under a second) |
| Write-path cost | None, but repeated scan load on reads | Extra writes inside every transaction | Near zero — the log exists anyway |
| Ordering guarantee | None across rows | Per-table, awkward across tables | Exact commit order |

Polling is the duct-tape version: cheap to build, but it misses deletes, misses intermediate states, and turns freshness into a scan-frequency knob you pay for in database load. Triggers fix completeness but put capture work inside the hot transaction path — every insert now performs two inserts, and trigger sprawl is notoriously hard to audit. Log-based CDC is where mature systems land, because the database already produces a perfect changelog for its own crash recovery and replication: Postgres's write-ahead log, MySQL's binlog, MongoDB's oplog. Reading it costs the primary almost nothing — it is the same interface a replica uses.

## Log-based mechanics

Every committed transaction is appended to the log before it is acknowledged; that is what makes the log authoritative. A log-based CDC connector — Debezium is the de facto open-source standard — connects like a replication client, decodes the log into row-level events (`op: c/u/d`, before-image, after-image, source position), and publishes them to Kafka, typically one topic per table, keyed by primary key.

That keying choice carries the ordering guarantee: Kafka preserves order within a partition, and all events for one key hash to one partition, so every consumer sees the changes to a given row in exactly the order the database committed them. Order across different keys is not guaranteed — and almost never needed. Broker internals belong to the Message Queues topic; what matters here is that per-key ordering survives the whole pipeline.

Two operational realities come with the territory. First, **snapshots**: a new pipeline (or a new consumer, like a freshly built search index) needs the current state, not just changes from now on, so connectors perform an initial consistent snapshot of each table and then switch to the log at the exact position the snapshot represents. Second, **schema evolution**: `ALTER TABLE` changes the shape of subsequent events, so production pipelines run a schema registry and choose compatible evolution (add nullable columns; never repurpose one) — otherwise a Tuesday migration breaks every consumer Wednesday morning.

## The outbox pattern

Raw CDC events are row-shaped: "row 42 in `orders` changed." Sometimes downstream services want intent, not rows — `OrderShipped`, with a specific payload, decoupled from your table layout. The transactional outbox delivers that without reintroducing dual writes: in the same database transaction as the business change, insert an event row into an `outbox` table. The transaction commits or rolls back atomically — business data and event together — and CDC tails the outbox table and publishes each row to Kafka. You get exactly the event you designed, with the atomicity of a local transaction. The outbox rows can be deleted immediately after insert; CDC reads them from the log, not the table.

## Delivery semantics

CDC pipelines are at-least-once, not exactly-once. A connector that crashes after publishing but before checkpointing its log position replays a window of events on restart. Consumers must therefore be idempotent: applying "set row 42 to shipped" twice is harmless, and upserts keyed by primary key make most CDC consumers naturally so. For consumers that count or append rather than overwrite, deduplicate on the event's log position (LSN / binlog offset), which uniquely identifies each change.

## Trade-offs and pitfalls

- **You are now running a pipeline.** Debezium, Kafka Connect, a schema registry, monitoring for connector lag — real operational surface. For one cache and one table, this can be overkill.
- **Replication slot leaks.** In Postgres, a stopped consumer with an open replication slot prevents WAL cleanup; the disk fills until the primary falls over. Alert on slot lag from day one.
- **Downstream is eventually consistent.** CDC lag is typically tens to hundreds of milliseconds, but spikes during bulk loads or migrations. Anything that must read its own writes should read the database, not the derivative.
- **Row events leak schema.** Publishing raw table changes couples every consumer to your table design. The outbox pattern (or a transform layer) is the fix; skipping it is the most common long-term regret.
- **Snapshots of big tables hurt.** Snapshotting a 2TB table can take hours and load the primary; schedule initial syncs off-peak and use incremental snapshotting where the connector supports it.

## When to reach for it

Reach for CDC the moment two or more downstream systems mirror the same database, or when you catch yourself writing "update X, then also update Y" in application code — that line is a bug you have not met yet. Cache invalidation feeds, search indexing, warehouse replication, and audit trails are the canonical wins, and Debezium into Kafka is the default stack unless your cloud's managed equivalent (like AWS DMS) already fits. Skip it when a single consumer can tolerate minutes of staleness — a cron job reading `updated_at` is honestly fine — and remember that CDC moves state changes; if what you need is commands or workflows between services, that is Message Queues territory, not a changelog.
