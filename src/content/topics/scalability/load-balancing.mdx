A single server has a hard ceiling — some number of requests per second beyond which latency explodes — and a guaranteed end: hardware fails, deploys go bad, kernels panic. A load balancer sits in front of a fleet of interchangeable servers, spreads requests across them, and stops sending traffic to any server that dies. It turns "a server" into "a service": capacity becomes a dial you turn by adding machines, and individual failures become invisible to clients. (When to grow the fleet is the Scaling topic's job.)

## Layer 4 vs layer 7

Where the balancer operates in the network stack determines what it can see, and therefore how smart it can be.

An **L4 balancer** works at the transport layer. It sees IPs and ports, picks a backend when a TCP connection (or UDP flow) arrives, then shovels packets without reading the payload. No parsing makes it extremely cheap — well under a millisecond of added latency, millions of packets per second per box. AWS NLB, Google Maglev, and Linux IPVS live here. The limitation: one connection maps to one backend for its whole life, and the balancer cannot route by URL, header, or anything inside the bytes.

An **L7 balancer** terminates the connection (including TLS), parses the protocol — usually HTTP — and makes a fresh decision per request. That buys routing by path or header, per-request balancing across a multiplexed HTTP/2 or gRPC connection, and retries against a different backend. nginx, HAProxy in HTTP mode, Envoy, and AWS ALB live here. The price is CPU for parsing and TLS, and a millisecond or two of added latency instead of microseconds. Large deployments stack them: an L4 tier for raw packet spreading, an L7 tier behind it for smart routing.

## Picking a server: the algorithms

Every algorithm answers one question — which backend gets the next request — but each optimizes for something different.

| Algorithm | How it picks | Optimizes for | Weak spot |
| --- | --- | --- | --- |
| Round robin | Next server in a fixed cycle | Simplicity; equal request counts | Ignores request cost and server speed |
| Weighted round robin | Cycle biased by static capacity weights | Mixed hardware generations | Weights go stale as reality drifts |
| Least connections | Server with fewest in-flight requests | Uneven request durations | Needs live counters; jittery at low traffic |
| Least response time | Fewest in-flight, tie-broken by recent latency | Latency-sensitive traffic | More state; reacts to measurement noise |
| IP hash / consistent hash | Hash of a client key maps to a server | Affinity and cache locality | Hot keys and NAT'd IPs skew load badly |
| Power of two choices | Sample two at random, take the less loaded | Near-least-connections balance at O(1) cost | Marginally worse than true least-connections |

**Round robin's blind spot is request cost.** Suppose 95 percent of requests are 10 ms reads and 5 percent are 2-second report generations. Round robin counts requests, not work: it keeps assigning traffic to a server already grinding through three reports, because "it's that server's turn." Queue depth on the unlucky server climbs, and tail latency for the whole service is set by the worst-balanced machine. **Least connections** fixes this with one cheap signal: in-flight count is a live proxy for busyness. A server stuck on slow requests holds connections open longer, its count stays high, and new traffic flows elsewhere — self-correcting without inspecting the requests at all.

**Power of two choices** is the trick worth memorizing. Plain random assignment balances poorly — spread n requests over n servers and the busiest ends up around log n / log log n above the average. Sample just *two* at random and take the less loaded, and the expected imbalance collapses to roughly log log n — near-perfect balance for one extra lookup. It also fixes a subtle failure of global least-connections: balancer instances sharing stale load data all pick the same "least loaded" server simultaneously and stampede it; two random choices decorrelate them. Hence Envoy's least-request policy samples two hosts by default, and nginx offers it as `random two least_conn`.

Hash-based affinity (IP hash, or a consistent hash over a request key) trades balance for locality — the same client or key always lands on the same server, which matters when that server holds a warm cache. The ring mechanics live in the Consistent Hashing topic.

## Health checks and draining

Routing around failure requires knowing what has failed. **Active health checks** probe each backend on an interval — say, hit `/healthz` every 2 seconds, mark the server down after 3 consecutive failures, up after 2 successes. The consecutive-count thresholds are hysteresis: one dropped packet should not flap a server out of rotation. **Passive checks** watch real traffic — connection refusals, timeouts, 5xx spikes — and eject misbehaving backends; Envoy calls this outlier detection. Passive detection reacts faster (a real user already hit the failure) but only fires under traffic; active probing catches dead servers even at 3 a.m. Production setups use both.

Removing a server gracefully is the other half. **Connection draining** (AWS: deregistration delay) means stop sending *new* requests to a server, let in-flight requests finish up to a deadline — typically 30 to 300 seconds — then take it out. Without draining, every deploy kills whatever requests were in flight, and users see errors on every release.

## Sticky sessions: the trade-off

If a server keeps session state in memory — a shopping cart, a login token cache — requests from that user must keep landing on that server. L7 balancers pin the user to a backend with a cookie; L4 balancers approximate it with IP hashing. This **session affinity** quietly forfeits much of what the balancer bought you: load skews (one NAT gateway fronting a large office looks like a single enormous client under IP hash), a crash loses every pinned session, draining slows to a crawl, and autoscaling half-works. The full cost accounting — and the escape, moving state off the instance so any server can handle any request — is the Stateful vs Stateless Architecture topic's story. Keep affinity only as a performance optimization (warm caches), never a correctness requirement — then losing a server costs cache misses, not user data.

## Global vs local balancing

Everything above balances within one site; getting users to the *right site* is a separate layer. **DNS-based** global balancing (GeoDNS, weighted records, health-checked failover as in Route 53) returns different IPs to different clients — cheap and universal, but failover is bounded by DNS TTLs, which many resolvers ignore, so a dead region can keep receiving traffic for minutes. **Anycast** announces one IP from many points of presence and lets BGP route each client to the nearest; failover is near-instant because routing simply converges elsewhere. Cloudflare and Google front their edges this way. A typical request thus crosses three deciders: anycast or DNS picks the region, an L4 tier picks a machine in the L7 fleet, and the L7 balancer picks the backend.

## Failure modes and pitfalls

- **Retry storms.** When backends slow down, clients and balancers retry — multiplying load precisely when capacity is scarcest, turning a slowdown into an outage. Cap retries with a budget (say, at most 10 percent extra load) and use jittered exponential backoff.
- **Thundering herd on recovery.** A server that just passed its health check gets a full traffic share instantly — with cold caches and connection pools, it may fall over again and flap. Slow-start ramps (HAProxy `slowstart`, Envoy `slow_start`) feed a recovering server a growing fraction of traffic over a warmup window.
- **Heterogeneous request costs.** If request costs span orders of magnitude, even least-connections struggles — split traffic classes onto separate pools, and watch per-server queue depth, not just request rate.
- **The balancer as single point of failure.** You have not removed the SPOF, you have moved it. Balancers run in redundant pairs (VRRP floating IPs) or behind anycast themselves; the redundancy arithmetic belongs to the Availability topic.
- **Health check lies.** A `/healthz` that only returns 200 says the process is up, not that it can reach its database. Shallow checks keep broken servers in rotation; overly deep checks let one dependency outage mark the whole fleet down at once.

## When to reach for it

The moment you run a second server, you need a load balancer — there is no sensible way to split traffic without one. In the cloud, take the managed option first: ALB (L7) or NLB (L4) on AWS and their equivalents elsewhere, so the balancer is someone else's availability problem. Self-managing, nginx or HAProxy covers almost every case; Envoy earns its complexity when you need gRPC-aware balancing, outlier detection, or a service-mesh data plane. For the algorithm, default to least-request with power-of-two-choices — essentially free and strictly better than round robin under real, uneven traffic. Reserve plain round robin for genuinely uniform workloads, hashing for cache affinity, and sticky sessions for legacy apps you cannot yet make stateless.
