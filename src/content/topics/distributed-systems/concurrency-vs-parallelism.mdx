A server juggling ten thousand open connections and a job multiplying two enormous matrices both "do many things at once" — yet they are bottlenecked in opposite ways and want opposite designs. Engineers who conflate concurrency and parallelism reach for threads when they need an event loop, or spin up an event loop when they need more cores. The distinction is the first fork in the road for almost every performance decision, from a single request handler to a fleet of machines.

## Structure vs execution: Pike's distinction

Rob Pike's formulation is the crispest one: **concurrency is dealing with many things at once; parallelism is doing many things at once.** Concurrency is a property of your program's *structure* — it is composed of independently progressing tasks that can be interleaved. Parallelism is a property of *execution* — at a given instant, more than one computation is physically running on more than one processor.

The two are orthogonal, and two examples pin down each corner:

- **Concurrent, not parallel:** a single-core machine running an event loop. Node.js on one core can hold 10,000 open sockets, interleaving work as each connection becomes readable. Only one instruction stream ever executes, but the program is structured around many simultaneous concerns.
- **Parallel, barely concurrent:** a GPU matrix multiply. Thousands of cores execute the same operation on different tiles of the matrix in lockstep. Enormous parallelism, but there is essentially one "thing" being dealt with — no independent tasks, no interleaving, no coordination logic.

## The decision rule: what is your bottleneck waiting on?

The distinction matters because the fix for a slow system depends on *why* it is slow.

| | I/O-bound work | CPU-bound work |
|---|---|---|
| Time is spent | waiting — network, disk, another service | computing — the CPU is pinned |
| What helps | concurrency: overlap the waits | parallelism: more cores or machines |
| Mechanisms | event loop, async/await, non-blocking I/O | threads, processes, worker pools, scale-out |
| More cores? | barely helps | roughly linear speedup |
| Examples | API gateways, proxies, chat servers | video encoding, ML inference, compression |

The intuition comes from the latency gap. A CPU executes an instruction in well under a nanosecond; a RAM access costs around 100 ns; an SSD read around 100 µs; a same-datacenter network round trip 500 µs or more. A request handler that calls a database spends 99%+ of its wall-clock time doing *nothing*. Parking that wait costs almost nothing if you do it right — which is the entire trick behind concurrency.

This is why **single-threaded Redis and Node.js are fast**. Redis serves hundreds of thousands of operations per second from one thread because each command touches in-memory data for microseconds; the expensive part is network I/O, which the event loop multiplexes across thousands of sockets via `epoll`. One thread also means no locks, no cross-thread context switching, and trivially atomic commands. The design stops working the moment a single command becomes CPU-heavy — a `KEYS *` scan over millions of keys blocks *every* client, which is exactly the failure mode the decision rule predicts.

## A tour of the models

**Shared-memory threads with locks** are the oldest model: threads share an address space and coordinate via mutexes. Both classic failure modes come from that sharing. A **race**: two threads run `counter += 1`, which is really read–modify–write; both read 41, both write 42, and one increment vanishes. A **deadlock**: thread 1 locks A then waits for B while thread 2 locks B then waits for A — both wait forever. Neither bug shows up reliably under test load, which is why the model has a reputation for 3 a.m. pages.

**The event loop** inverts the model: one thread, a queue of ready events, and non-blocking I/O. Concurrency without shared-state hazards — at the cost that any long computation on the loop freezes everything.

**Async/await** is the ergonomic layer over event loops. `await` marks the points where a task parks; the runtime resumes it when the I/O completes. You write sequential-looking code and get interleaving for free.

**Actors** (Erlang/Elixir, Akka) forbid shared memory entirely: each actor owns its state and communicates by asynchronous messages. Erlang's telecom heritage shows in the payoff — millions of lightweight processes, and one crashing actor cannot corrupt another's state.

**CSP** (Go) shares the "communicate, don't share" philosophy but uses channels between goroutines rather than mailboxes per actor. Go's slogan compresses it: share memory by communicating, don't communicate by sharing memory.

One language-level wrinkle worth knowing: CPython's **GIL** (global interpreter lock) lets only one thread execute Python bytecode at a time. Python threads give you concurrency — fine for I/O-bound work — but no parallelism for CPU-bound work; that requires multiple processes or native extensions that release the lock.

## Trade-offs and pitfalls

- **Concurrency adds coordination cost even without parallelism.** Interleaving means your invariants can be observed mid-update. A single-threaded Node service can still have "race conditions" across await points.
- **Blocking the event loop is the classic single-thread failure.** One synchronous 200 ms JSON parse adds 200 ms of latency to every in-flight request. CPU work belongs in worker threads or separate processes.
- **Threads are heavier than they look.** Each OS thread costs roughly 1 MB of stack and a context switch costs microseconds; 10,000 threads for 10,000 connections wastes gigabytes waiting. Goroutines and async tasks exist precisely because waiting should cost kilobytes, not megabytes.
- **Parallelism is capped by the serial fraction.** Amdahl's law: if 10% of the work is inherently sequential, infinite cores yield at most a 10x speedup. Measure before you shard the computation.
- **Locks compose badly.** Two individually correct lock-protected modules can deadlock when combined. Message-passing models trade this for a different pitfall: unbounded mailboxes and backpressure problems.

## The systems-level bridge

The same distinction scales up a level. A horizontally scaled service — 40 stateless replicas behind a load balancer — is **parallelism across machines**: many requests physically executing at once. Inside each replica, a request handler that fires three database queries and awaits them together is **concurrency**: one logical task structured so its waits overlap. Well-designed systems exploit both, and the sibling topics own the details — Scaling covers the machine-level story, Sync vs Async Communication covers the service-to-service one.

## When to reach for which

Profile first: is the workload waiting or computing? If it waits — on networks, disks, or downstream services — reach for concurrency: async/await or an event loop, and keep heavy computation off the hot path. If it computes, reach for parallelism: a worker pool sized near the core count, then more machines once one box is exhausted. Mixed workloads use both, in layers: nginx runs one event-loop worker per core, Node does the same via the cluster module, and Go schedules thousands of concurrent goroutines over a handful of parallel threads. And prefer models that avoid shared mutable state — channels, actors, or immutable messages — because the bugs you prevent are exactly the ones you cannot reproduce.
