A client sends "charge this card." The server charges it, then the response dies somewhere in the network. The client sees a timeout — and a timeout tells it nothing about whether the work happened. Its only safe move is to retry, so the same command arrives twice. Idempotency is the property that makes the second arrival harmless: performing an operation twice leaves the system in exactly the state that performing it once would.

## Doing it twice equals doing it once

Formally, an operation `f` is idempotent when `f(f(x)) = f(x)`. Concretely: `balance = 90` is idempotent — run it five times and the balance is 90. `balance -= 10` is not — five runs remove 50. The insight that turns this from math into design is that many operations can be *rephrased* from the second form into the first. "Deduct 10" becomes "set the balance to 90." "Append this event" becomes "insert this event with ID e-7842, if absent."

HTTP bakes this distinction into its verbs:

| Verb | Idempotent by contract? | Why |
|---|---|---|
| GET, HEAD | Yes | Reads change nothing |
| PUT | Yes | Full replacement — last write is the whole state |
| DELETE | Yes | Absent stays absent (the second call may 404, but state is identical) |
| POST | No | "Create a resource" done twice creates two |
| PATCH | Not guaranteed | "Set field x" is; "increment field x" is not |

Note what DELETE illustrates: idempotency is about resulting *state*, not identical responses. The second delete may return a different status code; the database looks the same either way.

Retries are not optional — every serious HTTP client, [load balancer](/SystemDesign/topics/scalability/load-balancing/), and queue consumer retries. So the real question for any mutating endpoint is: what happens when this arrives twice? For PUT-shaped operations, nothing. For "charge the card," "place the order," "send the email" — POST-shaped operations — you need to add idempotency deliberately.

## The idempotency-key pattern

The standard fix, with Stripe's API as the canonical design:

1. The **client** generates a unique key (a UUID) per *logical operation* — one key for "this checkout," reused verbatim across every retry of it — and sends it in an `Idempotency-Key` header.
2. The **server**, on first sight of the key, executes the operation and records the key together with the full response (status code and body), atomically with the side effect.
3. On any later arrival of the same key, the server skips execution entirely and **replays the stored response**. The retrying client cannot tell it was a replay — which is exactly the point.

Two failure modes make step 2's atomicity load-bearing:

**Partial failure mid-operation.** If the server charges the card, then crashes before recording the key, the retry re-charges. Write the key record and the side effect in the same [database transaction](/SystemDesign/topics/data-storage/acid-transactions/). When the side effect is a call to an external system, record the key in a "pending" state first, make the call passing your own idempotency key downstream, then mark it complete — a crash between steps is then detectable and reconcilable instead of silently duplicated.

**Concurrent duplicates.** A client whose timeout is shorter than your processing time retries while attempt one is still running. A read-then-execute check races: both requests see "key absent" and both execute. The fix is to make the key row itself the lock — insert it under a unique constraint before doing any work. The loser of the race either waits for the winner's result or gets an immediate 409, which is what Stripe returns for a concurrent reuse of a key.

Nuances that bite in production:

- **Key scope.** Scope keys per user or API credential, so one tenant's UUID collision can't replay another tenant's response.
- **Same key, different payload.** That's a client bug, not a retry. Store a hash of the request body with the key and reject mismatches (Stripe rejects these with an `idempotency_error`) rather than replaying an answer to a different question.
- **TTL.** Keys can't live forever, but expiring too soon reopens the duplicate window for slow retriers. Stripe prunes after 24 hours; pick a TTL comfortably longer than any plausible retry horizon.
- **Non-idempotent side effects inside the operation.** Recording the key protects the database write, not the confirmation email the handler fired along the way. Every downstream effect needs its own dedupe — typically by enqueueing it keyed by the same operation ID.

## Database tools that do the work

You rarely need to hand-roll the mechanics; databases ship idempotency primitives:

- **Unique constraints** — the bedrock. A unique index on the key (or a natural business key like `order_id`) turns a duplicate into a clean constraint violation you can map to "replay."
- **Upsert** — `INSERT ... ON CONFLICT DO NOTHING` (Postgres) makes "create if absent" a single idempotent statement.
- **Conditional writes** — DynamoDB's `ConditionExpression`, or compare-and-set on a version column, make "apply only if state is what I expect" atomic, which converts increments into idempotent transitions.
- **Idempotent consumers** — a queue worker stores each processed message ID in the same transaction as its state change, so redelivery becomes a no-op.

## The exactly-once myth

You will hear "exactly-once delivery" promised; over a lossy network it is impossible. If an acknowledgment can be lost, the sender must either retry (risking duplicates — at-least-once) or not (risking loss — at-most-once). Any system that cares about not losing data therefore chooses at-least-once — which is why [message queues](/SystemDesign/topics/architecture-patterns/message-queues/) and [webhook](/SystemDesign/topics/communication-apis/webhooks/) providers all deliver duplicates.

What *is* achievable is exactly-once **processing**: at-least-once delivery plus an idempotent handler equals effectively-once outcomes. Duplicates still arrive; they just stop mattering. Kafka's celebrated "exactly-once semantics" is this same move — producer sequence numbers for dedupe plus transactions — applied within its own ecosystem, not a repeal of the underlying physics.

## When to reach for it

First, try to make the operation naturally idempotent — prefer set over increment, PUT over POST, "insert with a caller-supplied ID" over "append." That's free and needs no protocol. Where the operation is irreducibly effectful — payments, order creation, provisioning, anything a user would notice happening twice — the idempotency-key pattern is mandatory, and it must be designed in from day one, because the client half (generate the key, reuse it across retries) can't be retrofitted server-side. For low-stakes internal writes like analytics counters, a rare duplicate may cost less than the machinery; decide explicitly rather than by accident. The test for any mutating endpoint you design: "what happens if this request arrives twice?" If the answer is anything other than "nothing," you have work to do.
